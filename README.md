# UE5资产加密方案评估与改进建议

## 问题背景与现状

用户在Unreal Engine 5中使用OpenSSL实现了资产加密机制，通过自定义 **IPlatformFile** 封装和部分定制 **FAssetTypeActions\_Base** 来对资产进行加解密。当前实现使得在**运行时**和**打包Cook**阶段可以正常使用加密资产。但在**编辑器**中发现：引擎的虚拟文件系统会缓存资产的解密内容，使得当进行复制、迁移、导出等操作生成新资产时，新的资产文件变成了**明文**（未加密）版本，从而可能泄露机密资源。

用户提出了两种解决方案来防止无密钥情况下资产内容外泄：

1. **复制/迁移/导出时重新加密新资产** – 在编辑器中执行复制、迁移、导出（会产生新资产文件）时，对新生成的资产再次执行加密存储。
2. **禁止加密资产的复制、迁移、导出、删除、移动等操作** – 从编辑器层面直接阻止对加密资产执行这些会产生明文输出的操作。

下面将分别评估这两种方案的可行性与局限性，并调研更优的替代方案，包括更底层的拦截机制或替代的文件访问封装方式。此外，将考察UE5中可用于拦截资产编辑器操作的API/机制，以及已有插件或案例可供借鉴。最后给出综合建议，确保加密机制对最终打包游戏资源同样有效。

## 方案一：操作时重新加密新资产

**实现思路：** 在资产复制、迁移、导出等需要生成新文件的操作时，拦截流程并对输出的资产文件重新执行加密。例如，当用户在内容浏览器中复制一个加密资产A为B时，确保B.uasset在保存到磁盘时仍以加密形式存储。潜在实现包括：监听资产保存事件，在检测到源资产是加密的情况下，对目标资产文件进行加密处理；或者定制资产复制流程，在复制时直接对数据进行加密写出。

**可行性：**
– Unreal引擎提供了一些针对资产保存和复制的接口和事件，可用于插入自定义逻辑。例如，保存包裹（UPackage）时引擎会调用`UPackage::SavePackage`，可以通过自定义PlatformFile或修改保存逻辑来影响文件写出。 引擎在保存资产时会对PackageFileSummary序列化三次，我们可以利用保存阶段在Summary序列化后修改输出数据，实现加密存储。另外，资产复制通常由AssetTools模块执行，例如`UAssetToolsImpl::DuplicateAsset()`，我们可以尝试在该流程结束后对生成的文件重新加密，或通过订阅`UPackage::PackageSaved`事件在资产保存完成后立即处理文件。理论上，通过这些钩子可以实现对复制/迁移/导出生成的新资产进行再次加密。

– **迁移（Migrate）**操作通常会将选定资产及其依赖项复制到另一项目目录。若迁移逻辑是简单的文件复制，则原始加密文件可直接拷贝（保持加密）。但若迁移通过加载后重新保存（用于更新引用路径），则需要类似于复制的拦截处理，确保保存时仍加密。总的来说，通过自定义保存流程，可以在**复制**和**迁移**的保存阶段重新加密新文件，使其保持与源资产相同的加密格式。

– **导出（Export）**操作（右键Asset Actions→导出）通常使用各资产类对应的UExporter将资产内容导出为外部格式（如静态网格导出FBX）。对于这类导出，如果目的是防止明文泄露，最简单可行的是**阻止导出**或对导出结果加密（但导出本质是为了在外部DCC工具使用，输出加密文件没有实际意义）。因此，在导出操作中，与其重新加密导出的FBX/PNG，不如直接禁用对加密资产的导出功能（方案二讨论）。

**局限性：**

* **实现复杂度高：** 针对每种操作需要深入不同流程添加钩子。复制和保存可以通过C++接口拦截，但需要确保时机正确。例如若在保存完成后加密，会有一个窗口期磁盘上存在明文文件；而在保存过程中加密则需要精细地处理引擎的序列化流程，否则可能破坏文件格式。需要非常了解UE5资产保存/复制机制。
* **覆盖面难以全面：** UE编辑器有多种生成新资产的途径，除了常规复制、迁移，还有可能用户通过**关卡转储**、**资产合并**、**重定向替换**等操作间接生成新资产。如果逐一拦截，容易有所遗漏。并且Blueprint/Python脚本也可调用复制功能（如`AssetTools.DuplicateAsset`），需要确保这些路径也被处理。
* **性能与稳定性：** 每次复制/保存都进行加密计算，会增加开销。对于批量迁移大量资产，加密过程可能显著延长操作时间。另一方面，修改引擎保存流程涉及底层字节操作，稍有不慎可能造成资产损坏或编辑器崩溃。
* **导出操作**难以用“重新加密”解决：因为导出目标一般是供引擎外使用的格式，无法期望外部工具支持解密。因此对导出最好还是直接禁止。

**小结：** 方案一在理论上**可行**，能够保持工作流的灵活性（允许复制等操作），但实现难度大且容易留下漏洞（某些未拦截路径导致泄密）。对导出操作而言，重新加密并不能真正保护资产（用户完全可以导出明文副本用于其他用途），因此实际意义不大。方案一需要非常深入且稳健的引擎改造才能彻底杜绝明文泄露，具有较高的开发和维护成本。

## 方案二：禁止敏感资产的编辑器复制/迁移/导出等

**实现思路：** 从编辑器层面限制对加密资产执行可能产生明文输出的操作。例如，当用户选中加密资产时，在内容浏览器的右键菜单中禁用或移除“复制（Duplicate）”、“迁移（Migrate）”、“导出（Export）”等选项；同时捕获删除、移动操作，对加密资产予以禁止（或弹出警告）。可以通过引擎的**内容浏览器菜单扩展**或**AssetTypeActions**/AssetDefinition机制来自定义这些操作的可用性。例如，为特定资产类型实现自定义的`FAssetTypeActions`，在`GetActions()`中不调用默认复制/迁移命令，或者返回`FCanExecuteAction`为false以禁用之。另一种方法是在编辑器启动时注册一个Content Browser菜单扩展，通过检查选中资产是否加密来有选择地禁用菜单项。

**可行性：**
– UE编辑器是可扩展的，**Editor Utility** (Blutility) 和 **Editor Module** 插件都可以对内容浏览器的行为进行定制。例如，通过\*\*内容浏览器菜单扩展(ContentBrowserMenuExtender)\*\*可以获得菜单构建器，对默认菜单项进行过滤或替换。也可以利用新版UE5的AssetDefinition机制（替代旧的AssetTypeActions）来自定义资产的行为（如是否支持拷贝）。因此，从技术上能拦截UI层面的操作入口。开发者可以编写插件，在编辑器初始化时插入钩子，检查选中对象列表，如果包含受保护资产，就禁用或移除相关菜单选项。

– 除了菜单，UE编辑器某些操作有对应的命令接口，可以在更底层禁止。例如**资产迁移**功能调用Editor资产工具模块，可以通过覆写或包裹`UAssetToolsImpl::AdvancedCopy`之类的方法来加入检查，检测到受限资产则直接中止操作。同时，也可以使用**EditorSubsystem**或全局委托监视资产操作事件：如删除资产时引擎会调用`UObject::Modify(true)`、`OnObjectPreDelete`等（具体需查UE5的编辑器事件），可在这些事件中拦截并取消操作。

**局限性：**

* **用户体验影响：** 完全禁止复制、迁移等，会让使用这些资产的开发者感到不便。无法复制资产意味着即使想基于现有资产做修改也不行；禁止迁移则无法将资源转移到其他项目或子项目中。在团队协作或资产迭代中，这可能带来流程障碍。
* **不一定防住高级方法：** UI层面的禁用可以防止普通用户误操作，但有经验的人仍可通过其它手段获取资产明文。例如直接在资源管理器复制.uasset文件到新项目（虽然拷贝的是加密文件，但如果连插件都一起复制过去并提供密钥，仍能打开）；或者编写Editor Utility脚本绕过UI调用引擎API复制资产。方案二主要防君子不防小人，对技术高手的恶意破解作用有限。
* **实现细节复杂：** 虽然禁用菜单看似简单，但需要覆盖编辑器各种交互：右键菜单、主菜单（如File->Export All之类），拖拽移动资产等。如果遗漏某个路径（比如通过重定向资产引用的方法复制），仍可能出现明文落地的情况。此外，完全禁止删除/移动可能没必要且增加负担 —— 删除加密资产通常不会造成泄密（删除仅移除加密文件），禁止删除反而影响正常管理。需要精细甄别每种操作的必要性。
* **维护成本：** UE版本升级可能改变编辑器菜单和命令实现，之前定制的禁用逻辑需要更新适配。尤其UE5在AssetDefinition方面有新改动，需跟踪文档以正确扩展。

**小结：** 方案二通过**限制功能**来杜绝明文泄露，从安全角度直接有效，但**代价是可用性的降低**。对于高度敏感的资产（例如付费受限资源），不让用户在编辑器中复制/导出可能是可以接受的 trade-off；但若这些资产需要正常参与开发流程，完全禁用会降低开发效率。该方案适合作为**辅助保护**（比如禁止导出、限制跨项目迁移），但不宜作为唯一手段，否则用户体验和工作流都会受到较大影响。

## 深层拦截与替代方案研讨

鉴于上述两种方案各有不足，我们调研了更底层的拦截机制和替代实现，以更全面地保护资产不以明文形式外泄，同时尽量减少对日常操作的干扰。

**1. 底层序列化层拦截（自定义保存/加载流程）**
最佳的方案可能是在**引擎资产序列化层**实现透明的加解密，使资产无论通过何种方式保存或复制，其落地文件始终是加密的。具体思路是**自定义UAsset文件格式或保存逻辑**：例如上文提到的Summary加密方案，在保存资产时拦截序列化流程，把关键元数据Summary和后续数据加密存储，并写入自定义的Magic Number标记。这样，任何通过`SavePackage`写出的资产都会自动套用加密格式，编辑器不会在磁盘留下明文。对应地，在加载流程中检查文件标记，识别出加密资产则读取加密块并解密，再交给引擎继续解析。这种改造深入引擎内部，一旦实现，**对所有资产操作均有效** —— 无论是用户手动复制、迁移，还是引擎自动保存，都将在底层统一加密处理。正如某案例所述，该方案能确保加密资产无法被未集成解密逻辑的引擎识别和打开，从根本上达成安全目标。

优点：

* **全面性：** 加密机制嵌入文件读写核心流程，杜绝任何绕过的保存途径。用户无需关心哪些操作可做哪些不可做，正常使用即可，安全策略在引擎底层自动执行。
* **透明性：** 对上层模块和用户而言，加密/解密过程是透明的，不需要手动触发。资产仍然以常规方式使用，差别仅在存储格式不同。
* **一致性：** 与打包流程更好融合。可以确保Editor阶段和Cook阶段采用相同或兼容的加密方案，最终打包的资源保持加密状态对运行时可解密（已验证该方案在Cook时也会经过自定义序列化流程）。

缺点：

* **需要引擎源码支持或扩展接口：** 这种改法通常需要修改引擎源码（例如重载CoreUObject保存读取代码）。如果无法修改引擎，也需要利用引擎提供的扩展点。例如通过自定义**Archive**或**FileWriter**插入加密步骤，但这要求UE5开放了相应钩子。实现难度高于简单封装PlatformFile。
* **复杂性与风险：** 重写资产格式需非常谨慎，以兼容UE本身对包结构的期望。一旦实现不当，可能导致资产损坏、版本兼容性问题等。此外，引擎层的改造可能与后续UE版本更新冲突，需要维护。
* **性能开销：** 序列化阶段对Summary和数据块的加解密会有一定性能损耗，但通常可以接受（且可只加密关键偏移信息以降低开销）。开发者应评估大量资产批量保存时的加密开销是否可容忍。

**2. 深度集成IPlatformFile到虚拟文件系统**
用户当前使用了自定义IPlatformFile来拦截文件读写，但出现编辑器缓存明文的问题。这提示可能需要将加密层集成得更深入或调整其工作方式。例如：

* **确保IPlatformFile同时处理读和写**： 当前明文泄露或许是因为写入新文件时未经过加密封装。如果尚未实现，对**FileWriter**的Write操作进行拦截加密是必要的。也就是说，自定义PlatformFile应包装底层文件系统，对于指定路径（如Content下.uasset/.uexp等）的写入数据，先执行加密再写磁盘。这样，无论编辑器以何种方式保存文件，都会通过我们的PlatformFile加密后存储，避免明文写出。
* **关闭或绕过引擎文件缓存**： 如果虚拟文件系统层仍缓存了解密后的数据（例如Pak文件系统会缓存已解密块以提高访问速度），需要在复制/保存时强制走一次实际的读取-加密-写入流程，而非直接拷贝缓存。可以考虑**不使用内存缓存副本**或者在保存前**清除缓存**，迫使系统重新从源加密文件读取原始二进制以保持加密性。具体需研究UE5虚拟文件系统的缓存机制。如果使用Pak虚拟文件，可在Mount时设置只读并禁止提取明文。
* **利用UE提供的虚拟化存储**： UE5引入了**Virtual Assets**机制，将大体积BulkData存储拆分，并支持Derived Data Cache等。可以考虑将Bulk数据加密存储于外部缓存，这样即使编辑器缓存元数据，真正的大数据部分仍是加密的。不过Virtual Assets偏重于版本控制优化，不直接提供安全保证，因此还需自行加密其数据。

**3. 利用Editor Utility或自定义资产编辑模块**
如果不修改底层，也可从编辑器工具层面增强对操作的管控，而不仅仅禁止。比如使用**Editor Utility Blueprint/Python**监听资产即将复制或导出的事件，提前做处理：取消操作或提示用户输入密钥确认。这虽然类似于方案二的思路，但可以做得更细腻，例如**仅限制受保护资产**（根据资产路径或标记判断），对于普通资产不干扰。UE编辑器并未公开明显的“OnAssetDuplicate”事件，但可通过间接途径实现：例如包装`UAssetToolsHelpers::GetAssetTools().DuplicateAsset()`函数，在蓝图Utility中调用前加入检查。另一个办法是编写**自定义AssetEditor**模块，当用户尝试打开或保存某类资产时接管流程，控制其后续操作。不过这种方式适用于特定资产类型（自定义UObject类），对UE已有类型资产不易全局套用。总体而言，Editor Utility更适合作一些**提示和辅助**（如在复制前警告），很难彻底拦截低层文件写入，安全性相对较弱。

**4. 使用受限的封装格式或权限**
除了软件方案，也可以考虑物理上限制资产的提取。例如将敏感资产打包成受密码保护的.pak文件或自定义容器，并**只挂载在编辑器内存中**。编辑器可以在启动时要求输入密钥挂载pak，内容读取时解密，这样在磁盘上始终只有加密pak存在。由于pak是整体加密的，用户无法单独复制某资产解密。但这种方式弊端是：编辑器对pak挂载内容的资产通常视作只读，可能无法直接在内容浏览器复制或修改，限制了资产的编辑能力。如果需求允许资产在编辑器中仅供查看使用、不做修改，那么通过加密Pak挂载是一种简便可行的保护方法（类似于DLC内容的加密分发）。Epic内置的pak加密功能就可用于打包发布阶段，但在编辑器开发阶段使用pak需要特殊处理流程。

**5. 参考已有插件或案例**
目前暂无公开的通用插件完全实现编辑器资产加密保护（大多讨论集中于**打包后**防提取，比如pak加密、修改AES密钥等）。然而，一些开发者分享的经验值得参考：有人建议**更改引擎对资产的处理细节**，例如改动资产文件格式Magic Number或加密方式，使现有解包工具无法直接识别（这正是前述Summary加密方案的思路）。还有安全领域的案例表明，即使加密也无法百分百防止破解（攻击者可以从内存提取密钥等），但**增加破解难度**是公认有意义的做法。因此我们在设计方案时，也应借鉴这些经验：采用非常规的加密格式组合、多重混淆来提高逆向成本。总之，虽然目前没有一键现成的插件，但社区观点和上述案例表明，**深入引擎实现定制加密**是可行且有效的，只是需要权衡开发投入和安全收益。

## 综合建议与最佳实践

综合以上分析，我们建议采取**更底层透明加密**为主，辅以**必要的操作管控**的方案，以实现安全性与可用性的平衡：

* **底层统一加密机制：** 优先实现对资产保存/加载流程的底层加密封装。理想做法是在引擎文件I/O或序列化层统一处理加密：例如参考Summary加密方案，在保存资产文件时自动写入加密内容，在加载时自动解密。这可以通过自定义IPlatformFile的完善实现（确保Write也加密），或修改引擎源码实现专用的加密Archive。这样，加密资产无论复制、迁移，其在磁盘上的表示始终是加密的，新生成的资产也自动继承加密格式，不会产生明文文件。此举也保证了打包到游戏时资源依然保持加密状态，未经授权无法解包查看。

* **操作拦截与限制：** 在底层加密到位的前提下，仍建议对某些高风险操作进行限制，以防止**合法用户**有意或无意输出明文副本：

  * **禁止直接导出明文：** 明确禁止对加密资产执行“导出”为引擎外格式的操作。这可以通过在AssetTypeActions或菜单扩展中移除导出选项实现。由于导出本质会产生明文副本，禁止之对大多数使用情景影响不大（授权用户通常也不需要导出这些受保护资产）。
  * **迁移提示或限制：** 若需要在受信任环境下跨项目迁移加密资产，应该提供专用的方法，而不使用编辑器默认迁移。可以开发一个**自定义迁移工具**：它在源项目中直接复制加密文件到目标，并确保目标项目也集成相应插件和密钥，从而仍保持加密保护。如果无法保证目标安全，则应禁止普通迁移，防止资产流出安全环境。
  * **复制操作视情况处理：** 在同一项目内复制资产可能是开发需要，应尽量允许。但要**确保复制出来的资产仍是加密存储**。如果底层机制已覆盖，那么允许复制是安全的（新文件已加密）。如底层机制一时无法完善，可选择暂时禁止敏感资产复制，以堵上明文漏洞，然后在机制完善后再开放该功能。

* **安全分层与密钥管理：** 无论采用何种加密方案，密钥管理是重中之重。建议使用**独立于项目源码的密钥存储**（如Editor启动时从安全存储加载密钥，而不是硬编码在插件中），并对密钥本身进行保护（例如仅授权机器/用户可获取）。同时，可以考虑对不同资产或模块使用不同的密钥、或结合硬件ID派生密钥，减少单一密钥泄露的风险。对于调试和开发，需要建立一套流程管理密钥（比如开发时使用测试密钥，发行时换正式密钥）。这些虽然超出纯技术实现，但直接关系到整体方案的有效性。

* **性能与兼容性考虑：** 加密措施难免带来开销，应测试在大量资产加载、保存情况下的性能影响。如果影响较大，可考虑**只加密关键部分**（如仅加密Summary/meta信息而非全部数据），以阻止未经授权的引擎识别资产即可。同时确保加密后的资源在Cook打包流程中不会干扰引擎的压缩和加载（通常引擎pak打包支持自定义加密块，只要我们遵循引擎机制即可）。对于已经加密的资产，建立检测机制避免重复加密（例如文件头含Magic Number时跳过），以免多次加密造成不可用。

**优缺点对比总结：**

| **方案**            | **优点**                                                             | **缺点 / 风险**                                             |
| ----------------- | ------------------------------------------------------------------ | ------------------------------------------------------- |
| **重新加密新资产** (方案一) | - 保持用户操作自由，复制迁移功能可用。<br/>- 理论上可使新资产继续保持加密。                         | - 实现复杂，需拦截多个流程。<br/>- 容易有遗漏导致漏洞。<br/>- 导出操作无法通过此方案安全支持。 |
| **禁止敏感操作** (方案二)  | - 简单直接，杜绝明文输出可能。<br/>- 实现相对独立，不需深改引擎。                              | - 严重影响工作流灵活性。<br/>- 只能防止误用，难防刻意绕过。<br/>- 后续维护成本随引擎变化增加。 |
| **底层加密集成** (推荐方案) | - 全面透明保护，各种操作下均无明文落地。<br/>- 对用户无感知，使用方式自然。<br/>- 与打包流程兼容，发行版同样受保护。 | - 开发量大，需深厚引擎知识。<br/>- 可能需要定制引擎或源码权限。<br/>- 初期性能开销需优化验证。 |

综上，**更底层的统一加密机制**搭配**有限的操作限制**是更优的选择。具体实施时，建议参考现有案例的技术思路，例如在资产保存时拦截Summary序列化并加密写入、在加载时识别自定义标记解密等，实现一个类似\*\*“加密资产文件系统”\*\*的模块。这样可以最大程度防止资产在未经授权的情况下以明文形式被提取或使用。同时辅以编辑器层面的安全策略（禁用导出、专用迁移工具等），防止合法用户的无意泄露。

最后，需要强调的是：**任何加密都有被破解的可能**，我们的目标是提高破解门槛。通过技术手段让资产保护更趋完善的同时，也要做好法律和管理上的配套措施。例如明确使用协议、对涉密资产进行水印标记等。双管齐下，才能最大限度地保障UE5项目资产的安全。




# 基于SharedKey和License的UE资产加密方案分析

## 方案概述与目标

在Unreal Engine编辑器中实现资产（如素材文件.uasset）的加密保护，用户提出一种以**单一对称密钥SharedKey**结合\*\*数字签名许可证（License）\*\*的机制来替代原先的“多用户密钥封装”方案。其主要目标包括：

* **统一加密**：所有受保护资产使用一个随机生成的共享对称密钥（SharedKey）加密（如AES算法），而不再针对每个用户生成独立的加密内容块，减少资产体积增长。
* **根密钥嵌入**：插件配置内置开发者的RSA公钥（`root_public.pem`）以及用于加密资产的SharedKey。开发者使用对应的RSA私钥（`root_private.pem`）签署许可证。
* **许可证授权**：由开发者使用`root_private.pem`对包含SharedKey和用户信息的**License文件**进行签名生成授权证书（本示例中使用自定义二进制格式）。只有持有有效License的用户才能通过插件验证，获得SharedKey来解密查看资产内容。
* **权限控制**：无License的用户虽然可以将受保护资产随项目一起运行游戏或打包发行，但**无法在编辑器中导出、迁移或复制资产内容**，确保资产不可被未授权人员提取出项目。

这一方案旨在在**不修改资产文件基本结构**的前提下，实现资产加密保护，使受保护资产可以被多个授权用户使用但无法被未授权用户导出。

## 使用流程与原理

该加密授权机制的工作流程如下：

1. **生成密钥并配置插件**：开发者随机生成一个对称加密密钥SharedKey（例如256位AES密钥），并准备一对RSA密钥对（root\_public.pem和root\_private.pem）。在开发者的UE插件配置中，内置root\_public.pem和SharedKey用于资源加密工作。

2. **资产加密存储**：当需要加密资产时，开发者通过插件对.uasset资源进行加密处理。具体而言，在保存资产到磁盘时，插件**拦截保存过程**，使用SharedKey对资产的关键数据部分（如文件Summary或数据段）进行AES加密。同时将.uasset文件头的标识Magic Number改为自定义的特殊值以标记其已加密状态。这样处理后，资产文件结构仍与普通uasset类似，仅在头部和数据段做了加密标记和密文存储，**保证未加密的资源仍可正常读取，不影响引擎兼容性**。未经授权的环境下，这种修改会使引擎无法识别该文件为合法资产，从而达到保护效果。

3. **发行许可证**：针对每一位授权用户或项目，开发者制作License文件作为访问凭证。License中可包含**用户标识**（如用户名、公司ID等）、**授权资产包ID**、**SharedKey（经过必要编码）**以及**有效期等信息**。随后对License内容使用`root_private.pem`执行数字签名，生成完整的许可证文件。数字签名确保许可证的来源可信且内容未被篡改。开发者将加密的资产文件和对应的License文件一并交付给用户。

4. **用户加载验证**：用户在Unreal Editor中安装含保护机制的插件（已内置root\_public.pem）。启动编辑器或加载项目时，插件会读取用户提供的License文件，并利用内置的RSA公钥验证其数字签名**是否由开发者私钥签发**。验证流程为：先用开发者公钥对License中附带的签名进行解密计算出哈希，再与License正文计算的哈希比对，若匹配则证明License确实由开发者签名且未被修改。只有签名合法的License才被接受，否则视为无效许可证，插件将拒绝提供解密权限。

5. **提取SharedKey**：签名校验通过后，插件从License中解析出SharedKey和相关授权信息（例如核对License中的资产包ID是否匹配当前资产）。插件在内存中缓存该SharedKey用于后续解密操作（并不会将其明文存储到磁盘）。由于软件**内置了开发者的公钥**而非私钥，无法由客户端伪造有效License。同时License中嵌入用户信息（如用户名、单位等）也使用户不愿意随意分享许可证，因为可以追溯到授权方。

6. **资产解密访问**：当授权用户在编辑器中尝试打开受保护资产时，UE引擎在加载该.uasset时首先读取文件头TAG。由于插件存储时写入了特殊Magic Number标记，插件的加载逻辑能够识别出该资产已加密状态。此时插件接管加载流程：**使用SharedKey对资产文件的加密数据段执行解密**，恢复出正确的资产数据，然后交由引擎继续正常的反序列化加载操作。解密过程发生在内存中，对用户透明且速度很快（AES对称解密开销小）。一旦解密成功，资产即可在编辑器中被查看和编辑，就像普通资产一样。整个过程仅在有有效License时才会进行；如果License无效或不存在，则插件无法获得正确的SharedKey，资产内容将保持加密状态，从而**在编辑器中无法被打开查看**。引擎会因为无法识别有效文件而报错或忽略该资产。

7. **运行与打包**：对于没有License的用户，虽然无法在编辑器查看资产细节，但仍**可以运行游戏或打包项目**而不解密这些资产。一种常见做法是将加密只限定在编辑器阶段，Cook打包过程中由插件提供解密，使得打包输出的游戏内容是明文资源，从而保证运行时正常使用。也就是说，在项目打包时，插件可以在后台使用SharedKey自动解密资产数据供打包器使用（前提是项目中存在有效License或内置密钥）。因此，未持有License的团队成员也能够编译运行项目，但依然无法从编辑器提取受保护资产。这满足了“多用户使用资产但防止未授权导出”的需求。需要注意，如果插件不对Cook流程做特殊处理，也可选择在发行版本中继续保持资产加密并在游戏运行时使用密钥解密（类似Pak文件加密机制），但那样密钥也需要随游戏发布且可能被专业工具提取。通常更安全的做法是在打包阶段就去除编辑期的加密，以免在最终发行物中藏有密钥。

## License文件设计与校验

**许可证文件**（License）是授权机制的核心。本仓库采用自定义的二进制格式，并在文件末尾附带RSA签名。为便于理解，下列仍以JSON示例展示其包含的字段：

```json
{
  "AssetPack": "UltraShaderPack",        // 授权资产包名称或ID
  "User": "StudioUserA (ID:12345)",      // 被授权用户或单位标识
  "ExpireDate": "2025-12-31",            // （可选）到期日期
  "SharedKey": "BASE64_ENCODED_AES_KEY", // 用Base64编码的AES密钥
  "Signature": "BASE64_SIGNATURE"        // 开发者对前面所有字段的RSA签名
}
```

上述示例中，Signature字段是对前面License数据（AssetPack, User, ExpireDate, SharedKey）的签名。实际签名流程为：开发者使用RSA私钥对License内容的哈希值进行加密得到数字签名。验证时，客户端插件用开发者公钥对该签名解密出哈希，并与计算得到的License内容哈希比对。若匹配则证明License未被篡改且确实由开发者签发，验证通过。验证代码可利用UE内置的Crypto库或第三方库实现，例如Crypto++在UE中就支持RSA签名/验签功能。

**License文件的校验逻辑**大致如下：插件读取License -> 提取签名字符串 -> 使用root\_public.pem验证签名是否匹配License正文 -> 根据License内信息（用户ID、资产ID、有效期等）判断授权范围。只有签名合法且授权信息匹配当前资产的情况下，插件才会接受该License并提取SharedKey用于解密。否则将拒绝授权并视同无License。整个验证过程发生在本地离线进行，无需联网验证，每次打开工程或加载License时执行一次，**性能开销很小**。

值得一提的是，License文件中包含的**用户身份标识和授权期限**等信息，除了在插件中用于信息比对外，也有助于增强安全性：例如记录授权用户姓名、单位等可以在一定程度上防止许可证被随意传播，因为许可证文件往往**嵌入了用户的身份信息，使其不太可能被用户自愿分享**。开发者也可以据此追查泄露来源或设置License过期需续期等策略。License中还可以加入其他策略字段（例如允许的功能列表、使用次数等），插件在验证通过后可据此启用或限制相应功能，类似软件授权管理。

## 可行性与安全性分析

**方案可行性**：从技术实现上，该SharedKey+License机制是可行的。Unreal引擎本身提供了数据加密和签名的基础设施（如AES加密库、Pak文件加密选项，以及Crypto模块）。插件能够借助UE的可扩展架构，在**不修改引擎源代码**的情况下挂接保存、加载流程，实现对资产文件的透明加解密。例如，正如某实践方案所述，可以在保存UAsset时**拦截PackageFileSummary的序列化**过程，将Summary数据加密存储，并在读取时检查自定义Magic Number后解密恢复Summary。这种方法只改变了文件头标识和数据存储方式，对资产文件结构影响很小，也不影响未加密的其他资产正常使用。因此，将其集成为UE编辑器的一个插件是方便的：插件可以注册Editor模块，在资源保存和加载点注入自定义逻辑，而项目内容创建和使用流程对美术或设计人员来说几乎无感知。

**安全性强度**：该方案结合了**对称加密**和**非对称签名**两种手段，各自承担不同职责：

* **SharedKey对称加密**保证了资产内容在存储层是保密的。AES等现代对称算法强度高，足以保护资产不被直接读取。指出UE内置AES加密采用CBC模式，可以避免简单模式下密文模式泄露统计信息，提高安全性。只要SharedKey不泄露，攻击者无法从加密文件直接恢复出明文资产。
* **RSA数字签名**保证了只有开发者签发的许可证才被承认。插件仅含公钥无法伪造签名，因此他人无法通过制作假License来骗取SharedKey。签名也防止了许可证文件被篡改（如更换用户ID或延长期限）。

在该机制下，**未获授权者几乎无法直接获取资产内容**：引擎在没有正确密钥时将无法读取加密资产的数据。即使尝试将受保护.uasset文件拷贝到未安装插件或无License的环境中，文件也因魔数不匹配等原因不会被识别为有效资产。正如资料所述，一旦资产经过此加密处理，在不支持解密的引擎上将“无法被识别为资源”，从而达到保护目的。

**与原方案安全性的比较**：相比原“多用户密钥封装方案”，新的License机制在管理上更简单但在单点泄露风险上有所不同。原方案可能为每个用户分别封装加密密钥（例如将内容密钥用每个用户的公钥加密封装，附在文件中），这样某一用户的私钥泄露不会直接危及其他用户的密钥。但实际上，如果任意一名授权用户解密并泄露了资产明文，本质上所有方案都会失去保护作用。因此，新方案采用单一SharedKey并不会明显降低整体安全：开发者可以通过License中的用户信息追踪泄露来源，而使用数字签名的离线许可证验证方式**避免了在线激活的繁琐，又能防止许可证被盗用或篡改**。

需要注意的是，**密钥管理**是安全性的重中之重：理想情况下SharedKey绝不能明文存储在客户端侧插件中，否则黑客可通过逆向插件提取该密钥，绕过License验证直接解密资产。为此，推荐将SharedKey**仅存于License文件**中，由开发者控制分发。插件本身只持有公钥，在没有License时并不包含任何可用于解密资产的敏感数据。这意味着无许可证用户即使拥有插件和加密资产，也无法自行解密。只有拿到开发者签发的许可证（其中含密钥且签名有效）才能解锁资产查看权限。这一点是方案安全性的基础。

**潜在攻击与对策**：和所有DRM机制一样，此方案并非绝对牢不可破，但足以防御大部分企图直接盗用资产的行为。攻击者可能的途径包括：尝试篡改引擎或插件绕过验证、内存Dump提取SharedKey、共享他人的License文件等。针对这些：

* RSA签名使篡改License或伪造授权基本不可能成功（没有私钥无法生成合法签名）。
* 插件可以在内存中对SharedKey做简单混淆，并在需要时才解密，以增加静态提取密钥的难度。同时，开发者可以选择定期更换SharedKey并更新资产和License，以缩短密钥泄露后的可用窗口。
* 对于**许可证文件的分享**问题，如果License未绑定具体机器，确实存在授权用户私下传播给他人的可能。开发者可以在License中加入硬件绑定信息（如机器ID、MAC地址等）并在验证时比对，以实现“一机一证”，但这种方式会影响使用便利性。另一种办法是在法律和协议层面约束用户不得私自分发License，并利用License内记录的用户身份来追责泄露者。
* 专业黑客仍可能通过逆向工程插件DLL来直接定位解密流程和密钥数据。但这种成本较高，而**我们通过自定义封装格式和加密手段，已大幅提高了破解门槛**。正如业内观点所言，“没有绝对安全，只能尽量增加逆向成本，当破解成本大于重制成本时，破解也就失去意义”。本方案正是通过修改资产格式和引入许可证校验，将随意导出资产的难度提高到了普通用户无法企及、黑客也需要付出相当努力的程度。

## 与原多用户密钥封装方案的比较

**原方案回顾**：多用户密钥封装通常是指为同一份加密内容针对不同用户分发不同的密钥封装。例如，每个授权用户都有各自的解密密钥，内容提供方可能将内容的对称密钥分别用每个用户的公钥加密，附加在同一个资产文件中，或者直接为每个用户生成一份专门加密给他的资产文件。这种方案确保只有特定用户能解开属于自己的那份密钥，从而访问内容。

相较而言，新方案通过**单一密钥+许可证**取代了逐用户封装，多用户授权管理方式发生了一些变化：

* **文件体积与分发**：原方案如果支持N个用户，可能需要在资产文件中附加N段封装后的密钥信息（或干脆生成N个不同版本文件），这会导致文件体积显著增加，并且分发管理繁琐。而新方案下，资产文件只需存储一次加密内容（和极少量加密标记元数据），无论授权给多少用户都不变；每个用户仅需持有一个小巧的License文件。这样内容分发上\*\*“一份加密资产文件 + 多个许可证”\*\*取代了多份冗余的加密文件，极大降低了存储和带宽开销。

* **密钥管理难度**：旧方案要求为每个用户维护一套密钥，对开发者来说管理多个密钥对、跟踪哪个用户对应哪把钥匙是额外负担。而新方案中开发者只需维护**一对根密钥**（用于签名）和一个内容SharedKey，发行许可证时也无需对内容重新加密，只需用私钥签名授权，**密钥管理更加集中和简化**。添加新授权只需签出新License，吊销授权则可以将其License列入黑名单或更新内容的SharedKey并重新加密发行新License，流程都比较直接。

* **多用户协作**：对于团队协作场景，新方案表现出优势。如果多个合法用户需要在同一个UE项目中使用同一资产，原方案可能需要在项目中存放多个不同加密版本（每人一份）或一个包含多人密钥封装块的文件，不仅麻烦还可能引入一致性问题。新方案下，团队成员各有各的License但引用的是**相同的加密资产文件**，项目文件无需重复，团队协作更便利。如果团队内部信任，共享一份License文件在多人机器上使用也是可行的（相当于项目级授权）。因此在**资产共享**方面，新方案避免了原方案逐人定制带来的障碍。

* **授权灵活性**：License机制可以灵活地在许可证中定义权限范围和有效期（例如只读权限、使用期限到某日期等），插件据此执行相应验证控制。相比之下，原多密钥方案通常只是简单地允许或拒绝访问，很难附加丰富的授权条件（除非借助额外的License文件配合，这又回到License机制本身）。因此新方案在授权策略上**可扩展性更强**。例如，可以发行仅限查看不可编辑的License，插件据此限制编辑操作，实现精细的权限控制。

* **安全风险对比**：原方案的优点是每个用户密钥相互独立，某用户密钥泄漏不直接影响其他用户。但从结果看，如果有任意一个用户泄露了解密后的资产，那么资产内容即被公开，其他方案也无计可施。因此在**抗内部泄露**上，两方案差异不大——都需要依赖法律协议和水印追踪等手段约束。新方案的劣势是**单点敏感**：SharedKey一旦被破解获取，所有资产保护失效。不过开发者可以通过更换SharedKey并重新发布更新来弥补，而且因为有License签名机制在，攻击者即使拿到SharedKey也无法制造假License去骗过插件对其他资产授权。原方案若某用户私钥外泄，同样会被他人利用来解密内容，危害也很大。因此综合看，两者安全性在实际应用中的差别更多体现在实现细节而非加密原理上的强弱。

* **实现复杂度**：原方案需要在资产文件格式中增加多段密钥封装数据或定制打包流程，对UE引擎的标准流程是更大的改动。而新方案充分利用了UE引擎的**可编程扩展**：通过插件层实现保存/加载时的加解密和授权验证，**侵入性较小**。开发者只需一次性开发和测试好插件及签发系统，即可反复使用，维护成本较低。原方案则可能需要为每次发货生成专门版本的资产并管理多版本文件，操作和维护都更繁琐。

综上，新方案在**效率、易用性和授权控制**方面相对于原多用户密钥封装具有明显优势，安全性也通过合理的密码学手段得到保障。唯一需要注意的是，集中式密钥意味着需要更严格地保护好SharedKey不泄露，但在整体风险可控范围内，这一方案大幅降低了运营管理复杂度，提升了资产发行的便利。

## 编辑器集成与操作拦截实现

在Unreal Editor中部署该加密授权机制，需要通过插件对某些编辑器行为进行Hook拦截，以防止未授权用户尝试通过常规功能提取资产。主要的拦截点和实现思路包括：

* **导出资产拦截**：Unreal编辑器允许用户右键资产选择“Asset Actions → Export”将资源导出为外部格式（如静态网格导出FBX，纹理导出PNG等）。插件可以拓展或替换导出逻辑，例如拦截 `IAssetTools::ExportAssets` 接口调用。具体做法是在插件启动时获取引擎的AssetTools模块，包装其导出函数：当检测到导出列表中包含受保护资产且当前无有效License时，阻止导出过程并提示“未经授权，无法导出此资产”。由于资产加密后在无License情况下本身数据不可读，这种拦截主要是防止出现不友好的错误提示，以及避免万一存在某些导出工具绕过了解密直接写出加密数据而导致困扰。通过Hook导出命令，可以在一开始就禁止未授权导出行为。

* **迁移资产拦截**：资产迁移（Asset Migrate）会将选定资产及其依赖拷贝到另一个项目。插件可拦截编辑器的迁移流程，例如监控`UEditorEngine::CopyAssets`或AssetTools的Migrate相关操作。当迁移操作涉及受保护资产且无License时，插件予以拦截，阻止这些.asset文件被复制到目标位置（或即使复制也因仍是加密状态对目标项目无用）。这样可避免未授权人员企图通过迁移功能将加密资产移出一个含插件的工程到不含插件的工程中去利用。另外，AssetRegistry扫描时加密资产因特殊魔数不会被识别为正常资源，即使强行拷贝文件过去，新项目也无法识别加载，这从文件格式上已形成了一道保护。

* **复制/粘贴和内容提取**：对于在编辑器中**复制资产内容**的行为，需要分情况考虑。若用户试图直接在内容浏览器中复制.uasset文件并在另一处粘贴，同样会遇到资产加密无法识别的问题，因此一般拦截不让直接复制文件即可（或让复制的仍是加密内容，对他人无意义）。如果是拷贝资产引用或在关卡中实例的对象，这并不会泄露资产内部数据，因而不必禁止。但对于**Blueprint蓝图等可见脚本类资产**，未授权用户打开时就被拒绝了，自然无法复制其中节点或逻辑。**素材内容**如纹理、模型，即使加载显示在Viewport中，插件也可以禁用截图、提取像素等调试命令（这一点需要深度定制，通常基础引擎不提供直接导出像素除非用户截屏）。总体来说，由于无License无法正常打开资产编辑器，大部分复制内容的途径已经被堵住。此外，可以考虑Hook编辑器的剪贴板复制事件，对于受保护资产执行复制命令时弹出警告或将剪贴板内容清空，从交互层面杜绝企图复制资产的操作。

* **AssetTools和Editor委托的利用**：Unreal引擎的模块化设计允许插件使用许多Editor委托（Delegates）来监控和修改编辑器行为。例如，可以订阅资产保存前后的事件、对象导出事件、文件菜单命令等。插件可以利用这些委托检测敏感操作。当发现未授权资产正要被保存到外部（导出或复制）时，立即中止操作。具体实现上，假如引擎没有提供直接的“导出前”委托，插件也可通过**扩展内容浏览器右键菜单**的方式，替换默认的“导出…”命令为自己的实现，在里面加入授权检查逻辑。对于AssetTools，我们可以派生定制一个子类注册为编辑器使用，从而override其中的导出和迁移方法。同样地，Editor层面的全局命令（如Ctrl+C复制、拖拽资产到操作系统窗口导出等）也可以通过捕获消息或Slate UI交互来封锁。通过多层次的Hook，确保未持有License的用户无论尝试何种方式，都无法顺利将资产内容导出到受保护环境之外。

需要注意的是，**这些Hook主要是增加额外的一道保障和提升用户体验**。即使没有这些拦截，只要加密本身可靠，未授权用户导出的文件也是加密无效的。然而，完善的拦截能够防止误操作造成加密资产的“损失”（例如某人不知情导出后发现文件不可用）、以及防止黑客通过自动化脚本批量尝试读取加密数据。因此，在插件集成上既要在底层加密解密环节做好不可绕过，又要在用户交互环节考虑周全，营造出资产\*\*“只能用不能偷”\*\*的安全使用环境。

## 优缺点分析

综上所述，采用SharedKey统一加密 + RSA签名License授权的方案，其主要**优点**和**潜在风险**可归纳如下：

**优点：**

* **单一密钥，加密效率高**：使用单个SharedKey对所有资产加密，避免了多重密钥封装带来的存储和处理开销。资产文件仅加密一次即可供所有授权用户使用，文件体积较小且加载解密开销低，不影响编辑器性能。
* **密钥管理简化**：开发者只需维护一把内容加密密钥和一对RSA密钥。授权新的用户时无需重新加密资产，只签发新的License文件即可；撤销用户时停止其License或更换SharedKey重新加密发行即可。密钥集中管理降低了出错概率和运营成本。
* **灵活的授权控制**：通过License文件可以细化授权策略，包括指定用户、使用期限、使用权限等。许可证签名机制确保这些授权条款不可被更改，插件可严格据此执行，满足不同发行场景需求（试用版、仅特定项目可用等）。
* **多用户协同支持**：多个用户可各持License共享同一份加密资产文件，无需重复存储。团队协作中只要有合法License，所有成员都能在项目中使用资产但无法单独导出，大大方便了合法使用场景。
* **不改变资产格式**：方案通过自定义MagicNumber等轻量方式标记加密状态，未对UE资产格式做颠覆性修改，确保**向下兼容**。未加密资产不受影响，加密资产在有插件环境下正常加载，在无插件环境下无法识别。这种非侵入式实现便于集成推广。
* **高安全性保障**：采用业界成熟的AES对称加密和RSA签名认证技术，安全可靠。只要开发者妥善保管私钥和SharedKey，外部几乎无法破解加密内容。即使有人获得加密文件，没有有效License也无法解密利用。整体安全性比不加密或简单钥匙机制有极大提升。

**潜在风险：**

* **单点密钥泄露**：由于所有资产共用一个SharedKey，一旦该密钥被攻击者获取，可能导致**所有受保护资产同时失守**。攻击者可能通过内存提取、插件逆向等手段尝试获取密钥。为降低风险，需在插件中尽量避免明文存放SharedKey，并可定期更换SharedKey发行更新。
* **许可证转借**：如果License未绑定具体用户环境，授权用户可能将自己的License文件分享给他人使用，绕过购买流程。尽管License嵌入了用户信息，这更多是软约束。技术上可以考虑加入机器绑定等措施，但会增加使用不便。如何平衡License灵活性与防盗用需要权衡，必要时结合法律手段和水印技术对付恶意分发。
* **对抗专业破解**：本方案主要防君子不防小人，能极大阻止普通用户乱拷贝资产，但面对专业黑客仍存在被攻破可能。例如，通过调试运行时内存或修改插件绕过校验。但这需要相当技巧和成本。开发者应认识到加密方案是**提高门槛**而非万无一失，一旦发行内容被恶意破解盗用，仍需依赖法律途径解决。
* **运行时密钥存在**：如果选择在游戏运行时才解密资产（而非打包时解密），则意味着在发行的游戏中仍然包含解密密钥和代码，可能被提取利用。因此一般应在发行版本中避免保留编辑器用的SharedKey，或使用独立的Pak加密体系并避免在内存中长时间存留明文密钥数据。
* **实现复杂度**：虽然方案主要在插件层实现，但开发这样的插件需要对UE资产序列化机制、编辑器扩展接口和加解密技术都有深入了解。一些低级别Hook（如拦截保存加载流程）实现不当可能引发兼容性问题。不过，一旦实现成功，后续维护工作量不大。
* **误操作与容错**：插件需要处理各种异常情况，例如用户没有License时尝试打开资产、License过期、签名验证失败等，避免出现崩溃或数据损坏。同时要确保即使插件或验证流程出现Bug，也不会错误地阻止正版用户使用或导致资产永久无法恢复（建议保留资产未加密的原始备份以防万一）。

总体而言，这套SharedKey+License资产加密授权机制在**便利性、安全性和灵活度**上较原有方案有显著提升。它能有效满足“资产可供多用户正常使用但不可被未授权者提取”这一需求，并且易于集成到UE编辑器工作流中。尽管存在一些需要注意的风险点，但通过合理的技术手段和管理策略，这些风险是可控的。对于希望保护数字资产的开发者来说，此方案具有很高的可行性和实用价值，可作为UE内容发行/分享时的一道重要安全防线。

# BSGuardEncryption

This Unreal Engine plugin transparently encrypts asset files using AES-256-GCM. A custom platform file intercepts reads and writes of `.uasset`, `.uexp` and `.ubulk` files so they are stored encrypted on disk.

## Setup
1. Build the plugin with your project and enable it.
2. In **Project Settings > GuardEncryption**, you can either paste a key string
   (`<Base64Key>|<Expiration>|<UserID>`) or select a signed **License file**.
   - Example key string: `AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=|2025-12-31T23:59:59Z|UserA`
   - License files are binary blobs generated by `ntag_sign.py` using the developer's private key and are typically stored as `*.license`.
3. Restart the editor to apply the key or license.

## Usage
* When a valid key or license is set, saving assets automatically encrypts them.
* Loading an encrypted asset requires a matching user ID and that the key or license has not expired.
* Expired licenses will be rejected; provide a new license or key string to continue using encrypted content.
